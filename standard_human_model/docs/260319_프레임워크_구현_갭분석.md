## 분석 목적

근육역학 인간모델 프레임워크 설계 문서에 명시된 요구사항과 현재 구현 상태를 항목별로 비교하여, 일치하는 부분과 차이가 있는 부분을 정리한다. 이를 통해 이후 개발 우선순위를 결정하는 기반으로 삼는다.

## 1. 잘 구현된 항목

### 1.1 핵심 7단계 파이프라인 (프레임워크 섹션 2.2)

프레임워크가 명시한 `pre_physics_step()` 내 7단계 흐름이 `human_body.py`의 `compute_torques()`에 정확히 반영되어 있다.

```
단계 1: q, dq 읽기             → moment_arm.compute_muscle_length/velocity()
단계 2: 척수 반사 레이어         → reflex_controller.py (stretch, GTO, reciprocal)
단계 3: 활성화 동역학            → activation_dynamics.py (1차 ODE, 비대칭 tau)
단계 4: R(q) → l_m, v_m       → moment_arm.py
단계 5: Hill 모델 힘 생성        → muscle_model.py (CE + PE + damping)
단계 6: τ = R^T @ F            → moment_arm.forces_to_torques()
단계 7: 토크 주입               → gym.set_dof_actuation_force_tensor()
```

실제로 `human_body.py:200~256`의 `compute_torques()` 구조가 이 흐름을 그대로 따른다. IsaacGym 통합 검증(I01~I04)에서 7번 토크 주입까지 확인 완료.

### 1.2 Hill 근육 모델 (프레임워크 섹션 2.5)

프레임워크가 지정한 수식 계보(Thelen 2003, Millard 2013)로 구현되어 있다.

- Active Force-Length: 가우시안 근사, width=0.45, 중심=l_opt (T01 PASS)
- Force-Velocity: 수축/신장 구간 분기 Hill curve, 신장성 최대 1.8배 (T02 PASS)
- Passive Force-Length: 이차 함수, l_opt 초과 시만 발생 (T03 PASS)
- Activation 선형성: active force = activation × (F_max × f_FL × f_FV × cos_penn) (T04 PASS)
- Pennation angle: cos(pennation) 항 포함

프레임워크가 기본으로 선택한 rigid tendon 가정도 일치한다. `k_tendon` 파라미터는 선언되어 있으나 현재 force 계산에서 사용되지 않는다 — rigid tendon 처리와 일관된 상태이다.

### 1.3 R 행렬과 Bi-articular 커플링 (프레임워크 섹션 2.6~2.7)

`moment_arm.py`의 `MomentArmMatrix`가 `(num_muscles × 69_DOF)` 행렬로 구현되어 있다. Bi-articular 근육 3종이 두 관절에 non-zero moment arm을 가진다.

| 근육군 | 걸치는 관절 | 커플링 효과 |
|--------|------------|------------|
| hamstrings_L/R | L_Hip + L_Knee | hip 신전 ↔ knee 굴곡 |
| gastrocnemius_L/R | L_Knee + L_Ankle | knee 굴곡 ↔ ankle 족저굴곡 |
| rectus_femoris_L/R | L_Hip + L_Knee | hip 굴곡 ↔ knee 신전 |

T05b 검증에서 이 6개 근육이 각각 2개 이상 관절에 0.1 Nm 이상 토크를 내는 것을 확인 (PASS). L/R 대칭도 완전 일치 (T06 PASS, 최대 오차 0.00 Nm).

### 1.4 반사 레이어 3종 (프레임워크 섹션 1.6, 3)

`reflex_controller.py`에 다음 세 반사가 구현되어 있다.

- Stretch reflex: `a_stretch = max(0, v_muscle - threshold) × stretch_gain` (시간 지연 버퍼 포함)
- GTO reflex: `a_gto = -gto_gain × max(0, F/F_max - gto_threshold)` (과부하 억제)
- Reciprocal inhibition: 주동근 활성화 시 길항근 억제

프레임워크가 명시한 경직/파킨슨/이완 환자별 파라미터 구분도 구현에 반영되어 있다.

| 환자 유형 | stretch_gain | stretch_threshold | reciprocal_gain |
|----------|-------------|-------------------|-----------------|
| Healthy | 1.0 | 0.10 | 0.3 |
| Spastic | 8.0 | 0.02 | (낮음) |
| Flaccid | 0.0 | 999 | - |

IsaacGym 검증에서 Spastic 82.4° > Healthy 74.4° > Flaccid 23.0° 확인 (I03 PASS). 반사 토크 비율 |Spastic|=1.95 Nm >> |Healthy|=0.15 Nm (I04 PASS).

### 1.5 인대/관절낭 모델 (프레임워크 섹션 3)

`ligament_model.py`에 지수적 soft limit 토크가 구현되어 있다. 프레임워크 수식과 일치한다.

```
τ_ligament = -k_lig × exp(alpha × max(0, q - q_soft_upper))
           + k_lig × exp(alpha × max(0, q_soft_lower - q))
```

ROM 경계에서 부호 방향과 활성화 구간 확인 완료 (T07 PASS).

### 1.6 환자 프로파일 시스템 (프레임워크 섹션 5)

YAML 파라미터만 교체하여 환자군을 전환하는 구조가 구현되어 있다.

| 파일 | 대상 환자 |
|------|---------|
| `profiles/healthy/healthy_adult.yaml` | 정상 성인 |
| `profiles/stroke/stroke_r_hemiplegia.yaml` | 뇌졸중 우편마비 |
| `profiles/parkinson/parkinson_moderate.yaml` | 파킨슨 중등도 |
| `profiles/sci/sci_t10_complete_flaccid.yaml` | 척수 완전 손상 (이완) |
| `profiles/sci/sci_incomplete_spastic.yaml` | 척수 불완전 손상 (경직) |
| `profiles/cp/cp_spastic_diplegia.yaml` | 뇌성마비 양측 경직 |

프레임워크 섹션 1.7의 병리-파라미터 매핑 표와 일치한다.

### 1.7 검증 구조 수준 1~2 (프레임워크 섹션 6.1~6.2)

| 검증 수준 | 내용 | 상태 |
|----------|------|------|
| 수준 1 — 구성요소 | T01~T09 (Hill, Moment Arm, Ligament, Reflex, Pipeline) | 9/9 PASS |
| 수준 2 — 현상 | I01~I04 (IsaacGym 통합, 환자 분화, 반사 차등) | 4/4 PASS |

## 2. 프레임워크와 다른 항목 (Gap)

### 2.1 l_opt 단위 혼용 — 가장 시급한 수정 필요

**프레임워크**: `l_opt`는 실제 최적 근섬유 길이(미터 단위). quadriceps 0.08m, hamstrings 0.10m, gastrocnemius 0.05m, soleus 0.04m.

**현재**: `healthy_baseline.yaml`에서 모든 근육의 `l_opt=1.0` (무차원). `l_slack`은 미터 단위(0.30m)이므로 `l_norm = 0.30 / 1.0 = 0.30`이 된다.

**영향**:
- 이 동작점에서 active force = F_max × f_FL(0.30) ≈ F_max × 0.30 → 최대 힘의 30%만 발휘
- passive force는 l_norm ≤ 1.0이므로 발생하지 않음 (과신전 상황 제외)
- 근육이 실질적으로 약화된 상태로 동작하고 있음

**수정 방향**: 각 근육의 `l_opt`를 해부학적 실측값(미터)으로 교체. 수정 후 T01 플롯의 동작점이 1.0 근처로 이동하고 passive force도 과신장 시 발생한다.

코드 버그가 아니라 YAML 파라미터 이슈이므로 코드 수정 없이 파라미터 파일 수정만으로 해결 가능하다.

### 2.2 아킬레스건 Elastic Tendon(SE) 미구현

**프레임워크 섹션 2.5**: 가자미근(soleus)과 비복근(gastrocnemius)에 한해 elastic tendon 구현을 강하게 권장한다. 보행 에너지 효율의 35% 이상이 아킬레스건의 탄성 반동에서 기인하기 때문이다.

```
F_SE = k_tendon × ((l_tendon - l_slack) / l_slack)²
```

**현재**: rigid tendon 가정이 모든 근육에 일괄 적용된다. `HillMuscleModel.compute_force()`에서 SE 계산 코드가 없고, `k_tendon` 파라미터는 선언되어 있지만 사용되지 않는다.

**영향**: push-off 단계에서 아킬레스건의 탄성 에너지 저장/방출이 재현되지 않아 보행 에너지 효율이 실제보다 낮게 시뮬레이션된다. 보행 제어기 통합 이전에 구현하는 것이 권장된다.

### 2.3 R(q) 상수 행렬 (알려진 한계)

**프레임워크**: 중장기 개선 과제로 각도 의존 R(q) 언급.

**현재**: `MomentArmMatrix`는 상수 R을 사용한다. `compute_muscle_length()`에서 `dof_pos` 인자를 받지만 실제로 사용하지 않는다(`# 현재는 미사용 (상수 R)` 주석 존재).

**영향**: 무릎 굴곡 각도에 따라 hamstrings의 moment arm이 20~30% 변하는 효과가 없다. 보행 중 자세 변화에 따른 토크 생성 정밀도가 저하된다.

검증 문서에 이미 명시된 known limitation이다.

### 2.4 하지 근육군 매핑 차이

**프레임워크 섹션 2.7**: 한쪽 하지 기준 11개 근육군을 명시한다.

| 프레임워크 명칭 | 현재 구현 | 상태 |
|--------------|---------|------|
| 장요근 | hip_flexors | 통합 ✅ |
| 대둔근 | gluteus_max | ✅ |
| 중둔근 | hip_abductors | 통합 ✅ |
| 내전근군 | hip_adductors | ✅ |
| 대퇴직근 | rectus_femoris | ✅ (bi-articular) |
| 내/외측광근 | quadriceps | 통합 ✅ |
| 햄스트링 | hamstrings | ✅ (bi-articular) |
| 비복근 | gastrocnemius | ✅ (bi-articular) |
| 가자미근 | soleus | ✅ |
| 전경골근 | tibialis_ant | ✅ |
| **장지굴근** | **없음** | **❌ 미구현** |

장지굴근(flexor digitorum longus)은 발목-발가락 bi-articular 근육으로, 프레임워크 R 행렬에 ankle과 foot DOF에 비영 값이 있다. 발가락 굴곡과 발목 족저굴곡에 동시 기여하는 역할을 한다. 현재 구현에서 빠져 있다.

### 2.5 검증 수준 3~4 미완 (프레임워크 섹션 6.3~6.4)

| 검증 수준 | 내용 | 상태 |
|----------|------|------|
| 수준 3 — 규명 교차 검증 | 부분 실험으로 규명한 파라미터가 보류 조건을 예측하는지 | ❌ 미완 |
| 수준 4 — 통합 보행 검증 | 보행 제어기 연결, 시공간 파라미터, 지면반력 패턴, 보조기기 상호작용 | ❌ 미완 |

현재 검증은 단일 관절(L_Knee) 수준의 진자 실험까지만 완료된 상태다. 전신 보행 제어기와의 통합은 아직 시작되지 않았다.

### 2.6 Real-to-Sim 파라미터 규명 파이프라인 미구현 (프레임워크 섹션 4)

**프레임워크**: 임상 측정 → 파라미터 규명의 5단계 순차적 파이프라인.

1단계: 수동 ROM 검사 → k_PE, ε₀, k_ligament 규명
2단계: 등속성 근력검사 → F_max, v_max, A_f 규명
3단계: 급속 수의 수축 → τ_act, τ_deact 규명
4단계: 진자/교란 검사 → g_reflex, v_threshold 규명
5단계: 보행 분석 → 통합 검증

**현재**: 모든 파라미터가 문헌 초기값(Thelen 2003, Rajagopal 2016)으로 설정되어 있다. 실제 임상 데이터를 받아 파라미터를 피팅하는 코드는 존재하지 않는다.

이 파이프라인의 구현은 실제 환자 데이터가 확보된 시점에 필요하다. 현재 단계에서는 우선순위가 낮다.

## 4. 다음 단계 권장 순서

우선순위는 다음 기준으로 결정했다. 보행 제어기 통합 실험(VIC-MSK)이 가까운 시일 안에 예정되어 있으므로, 그 이전에 모델의 물리적 정확도에 직접 영향을 주는 항목을 먼저 처리한다.

**1순위 — l_opt 파라미터 수정 (코드 수정 불필요)**
현재 근육이 동작점 30%에서 작동하고 있다. YAML 파일에서 각 근육의 l_opt를 실측 미터 단위로 교체하면 active/passive force가 생리적으로 정확한 동작점에서 발휘된다. 코드 변경 없이 파라미터 파일 수정만으로 해결 가능하다.

**2순위 — 아킬레스건 Elastic Tendon 구현**
보행 push-off 에너지 효율에 직접 영향을 준다. `HillMuscleModel`에 가자미근/비복근에 대한 SE 계산 분기를 추가하고, `healthy_baseline.yaml`에 `l_slack_tendon`과 `k_tendon` 실측값을 설정한다.

**3순위 — 검증 수준 4 (보행 통합 검증)**
보행 제어기와 연결 후 관절 토크 RMSE, 시공간 보행 파라미터, 지면반력 패턴을 검증한다.

**4순위 — 장지굴근 추가**
발가락 굴곡과 발목 족저굴곡 커플링 재현을 위해 추가한다. push-off 정확도 향상에 기여하나, 우선순위는 위 항목들보다 낮다.

**5순위 — R(q) 각도 의존성, Real-to-Sim 규명 파이프라인**
중장기 개선 과제. 실제 환자 데이터 확보 후 구현.
