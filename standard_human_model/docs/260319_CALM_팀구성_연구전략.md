# CALM 프레임워크 팀 구성 및 연구 전략

작성일: 2026-03-19
인원: 4~5명 (PI 포함)

---

## 1. 전체 구조 한눈에

```
┌─────────────────────────────────────────────────────────────┐
│                    CALM 프레임워크 스택                        │
│                                                             │
│  [Layer 4] 상위 제어기 (RL / CPG / 임피던스 제어)             │  ← Role D
│  [Layer 3] 인간-로봇 통합 시뮬레이션 (Isaac Lab)              │  ← Role C
│  [Layer 2] 환자별 파라미터화 (Real-to-Sim 파이프라인)          │  ← Role B
│  [Layer 1] 근골격 역학 코어 (Hill + Reflex + Ligament)        │  ← Role A
│  [Layer 0] 인간 신체 모델 (SMPL/URDF + Exo 에셋)             │  ← Role A+C
└─────────────────────────────────────────────────────────────┘
```

각 레이어는 명확한 입출력 인터페이스로 분리되어 있어, 담당자가 다른 레이어 코드를 알 필요 없이 독립 개발이 가능하다.

---

## 2. 역할 정의 (4인 기준)

### Role A — 근골격 역학 코어 (Biomechanics Lead)
**담당**: PI (현재 연구자, VIC + PHC 배경)

**이유**: 이 레이어는 생체역학 수식의 정확성과 코드 구조 전체가 연결되는 허브다. PHC/VIC 경험에서 pre_physics_step 구조를 이미 이해하고 있고, 프레임워크 설계 의도를 가장 잘 알고 있는 사람이 맡아야 한다.

**담당 태스크**:
- `core/muscle_model.py` 고도화: l_opt 단위 수정, 아킬레스건 elastic tendon (SE) 구현
- `core/moment_arm.py` 고도화: R(q) 각도 의존성 (polynomial fitting)
- 프레임워크 전체 인터페이스 설계 및 유지 (API 계약 문서 작성)
- 검증 수준 3~4 설계 및 감독
- 다른 Role의 PR 리뷰

**주요 산출물**:
- `core/` 전 모듈 (muscle_model, moment_arm, activation_dynamics, reflex, ligament)
- `config/` YAML 스키마 정의
- 기술 문서 및 API 계약

---

### Role B — 환자 파라미터화 (Clinical Parameterization)
**담당**: 임상/재활공학 또는 생체역학 배경 1인

**이유**: 임상 측정 프로토콜을 이해하고 환자 데이터를 파라미터로 변환하는 파이프라인은 생체역학 + 임상 지식이 동시에 필요하다. Role A가 코어 수식을 담당하므로, Role B는 그 위에서 파라미터 수치를 채우는 역할에 집중할 수 있다.

**담당 태스크**:
- Real-to-Sim 규명 파이프라인 구현 (4단계: 수동역학→최대힘→활성화→반사)
- 환자 프로파일 YAML 확장 및 검증 (stroke, Parkinson, SCI, 노인, 근력약화)
- 임상 측정 데이터 수집 프로토콜 문서화
- 문헌 기반 파라미터 근거 정리 (Rajagopal 2016, Thelen 2003 등)
- 검증 수준 3: 훈련-검증 분리, 교차 양식 예측

**주요 산출물**:
- `tools/param_identification/` — 등속성 검사, 진자 검사 데이터에서 파라미터 피팅 스크립트
- `profiles/` — 환자 프로파일 YAML (최소 4개 질환군 × 경증/중증)
- `docs/clinical_measurement_protocol.md` — 임상 측정 가이드

---

### Role C — 로봇 모델 및 통합 환경 (Robot Integration)
**담당**: 로보틱스/URDF 경험 1인

**이유**: Exoskeleton URDF 설계, Isaac Lab 마이그레이션(PhysX 5), closed kinematic chain 처리는 Role A~B와 독립적으로 병렬 진행 가능한 영역이다. IsaacGym/Isaac Lab API와 물리 엔진 설정에 대한 이해가 필요하다.

**담당 태스크**:
- Exoskeleton URDF/MJCF 설계 (무릎형/발목형 보조기 우선)
- IsaacGym → Isaac Lab(PhysX 5) 마이그레이션 (closed kinematic chain 지원)
- 인간-로봇 체결부 모델링 (스트랩 constraint, contact surface)
- 상호작용력 센서 시뮬레이션 (virtual load cell)
- `isaacgym_integration/` → `isaaclab_integration/` 재구성

**주요 산출물**:
- `assets/exo/` — Exo URDF 에셋 (무릎형, 발목형)
- `envs/human_robot_env.py` — 인간+로봇 통합 IsaacLab 환경
- `docs/robot_integration_guide.md`

---

### Role D — 상위 제어기 (Control & Learning)
**담당**: RL/제어 배경 1인

**이유**: VIC 연구에서 이미 rl-games + AMP 파이프라인이 구축되어 있다. Role A~C가 제공하는 `compute_torques()` API만 사용하면 되므로, CALM 코어 내부를 깊이 알 필요 없이 제어기 설계에 집중할 수 있다.

**담당 태스크**:
- 보행 모방 + CALM 근골격 레이어 결합 RL 학습
- CPG (Central Pattern Generator) 기반 보행 제어기
- 환자 파라미터별 보조 전략 비교 실험
- 상위 제어기 벤치마크 구성 (정상인 기준선 → 환자 프로파일 전환 실험)
- Sim-to-Real 갭 분석

**주요 산출물**:
- `controllers/rl_policy.py`, `controllers/cpg_controller.py`
- `experiments/` — 실험 설정 스냅샷
- 학습 결과 분석 문서

---

### 5인일 경우: Role E — 검증 및 시각화 (Validation & Visualization)
**담당**: 데이터 분석 또는 대학원 초기 과정 인력

Role A~D 모두에게 검증 코드 작성은 부담이다. 별도 인력이 독립적으로 검증 스크립트를 작성하면 각 레이어의 품질 보증이 더 체계적으로 된다.

**담당 태스크**:
- 수준 3~4 검증 스크립트 작성 및 실행
- 보행 분석 시각화 (관절 각도, 토크, GRF, 근활성화)
- 환자 프로파일 비교 플롯 자동화
- `validation/` 전 단계 유지보수

## 3. 기술 인터페이스 계약

각 Role이 독립 개발 가능하려면 레이어 간 입출력을 미리 고정해야 한다. 다음이 변경 금지 인터페이스다.

### Interface 1: CALM Core API (Role A → 모두)

```python
# HumanBody.compute_torques() 시그니처 — 변경 금지
def compute_torques(
    dof_pos: Tensor,          # (num_envs, 69) — IsaacGym/Lab DOF 상태
    dof_vel: Tensor,          # (num_envs, 69)
    descending_cmd: Tensor,   # (num_envs, num_muscles) [0, 1] — 상위 제어기 출력
    dt: float,
    contact_forces: Tensor = None,  # (num_envs, 4) 발바닥 접촉력 (선택)
) -> Tensor:                  # (num_envs, 69) 관절 토크
```

Role B는 이 API의 파라미터 YAML만 교체한다.
Role D는 `descending_cmd`를 출력하는 제어기만 만들면 된다.
Role C는 이 토크를 `set_dof_actuation_force_tensor()`에 주입하는 환경 코드를 작성한다.

### Interface 2: 환자 프로파일 스키마 (Role B → Role A)

```yaml
# profiles/[condition]/[name].yaml 스키마 — 변경 금지
name: str
patient_type: str
muscle_params:
  [muscle_name]:
    f_max: float   # N
    l_opt: float   # m (미터 단위 강제)
    v_max: float   # l_opt/s
    ...
reflex_params:
  default: {stretch_gain, stretch_threshold, gto_gain, gto_threshold, reciprocal_gain}
ligament_params:
  {k_lig, alpha, damping, soft_limit_margin}
```

Role A가 스키마를 정의하면 Role B는 그 스키마 안에서 수치만 채운다.

### Interface 3: 로봇 통합 환경 API (Role C → Role D)

```python
# HumanRobotEnv.step() — 변경 금지
def step(
    action: Tensor,           # (num_envs, action_dim) — 상위 제어기 출력
) -> Tuple[Tensor, Tensor, Tensor, dict]:
    # obs, reward, done, info
    # obs에는 dof_pos, dof_vel, bio_torque, exo_torque, contact_force 포함
```

Role D는 환경 내부(URDF, 물리 설정)를 몰라도 된다.

## 4. 통합 전략

### 4.1 브랜치 전략

```
main          — 통합 완료, 항상 동작 보장
develop       — 통합 진행 중
feat/core-*   — Role A 작업 (core/ 수정)
feat/patient-*— Role B 작업 (profiles/, tools/)
feat/robot-*  — Role C 작업 (assets/, envs/)
feat/ctrl-*   — Role D 작업 (controllers/, experiments/)
```

`develop → main` 머지는 PI(Role A)가 승인. 각 feat 브랜치는 자유롭게 커밋.

### 4.2 통합 체크포인트 (Phase별 합류 시점)

**Phase 1 — 독립 개발 (약 6주)**

| Role | 목표 |
|------|------|
| A | l_opt 수정 + elastic tendon + R(q) 1단계 |
| B | Real-to-Sim 파이프라인 설계 + 2개 환자 프로파일 완성 |
| C | Exo URDF v1 + Isaac Lab 환경 구축 |
| D | 기존 VIC 정책으로 기준선 실험 |

**Phase 2 — 레이어 통합 (약 4주)**

- Milestone 1: Role A의 고도화된 CALM Core + Role B의 환자 프로파일 → 검증 수준 3 실행
- Milestone 2: Role C의 통합 환경 + Role A의 Core 연결 → 인간+로봇 물리 시뮬 동작 확인
- 통합 테스트: `integration_test/run_all.py` 실행이 PASS여야 develop→main 허용

**Phase 3 — 학습 및 검증 (약 8주)**

- Role D가 통합 환경 위에서 상위 제어기 학습
- Role B가 환자 파라미터별 비교 실험 설계
- Role E(있다면)가 전체 검증 수준 4 실행

### 4.3 인터페이스 변경 프로세스

Interface 1~3은 변경 시 반드시 PI 사전 승인이 필요하다. 변경이 필요할 경우:
1. `docs/interface_change_request/YYMMDD_변경내용.md` 작성
2. 영향 받는 Role 담당자들과 동기화 회의
3. 변경 전 기존 시그니처를 deprecated로 유지 (1 Phase 동안)

### 4.4 주간 동기화

매주 1회 30분 체크포인트 미팅. 형식:
- 각자 3줄: 지난 주 완료 / 이번 주 목표 / 블로커
- 인터페이스 관련 이슈만 전체 논의, 나머지는 1:1로

## 6. 논문 전략

### 6.1 이 프레임워크 자체로 쓸 수 있는 논문

#### 논문 A — 메인 시스템 논문 (Systems/Methods paper)
**제목 방향**: "CALM: A Clinically Observable Musculoskeletal Layer for Patient-Specific Wearable Robot Simulation"

**타겟 저널/학회**: IEEE TNSRE (Trans. Neural Systems & Rehabilitation Engineering), ICRA, RA-L

**핵심 주장 (Claim)**:
임상에서 비침습적으로 측정 가능한 파라미터만으로 구성된 관절 수준 근골격 동역학 레이어를, GPU 병렬 RL 환경(IsaacGym) 안에서 구현할 수 있으며, 동일한 구조에서 파라미터 변경만으로 뇌졸중/파킨슨/SCI 등 다양한 환자군의 병리적 거동을 재현할 수 있다.

**논문 구성**:
1. Introduction: 기존 근골격 시뮬(MyoSuite)의 파라미터 관측 불가능성 문제, 토크 직접 지정(IsaacGym PD)의 한계
2. Framework: 7단계 파이프라인, Hill 모델, R(q), 반사 레이어, 인대 모델
3. Parameter Observability: 각 파라미터가 어느 임상 검사로 규명 가능한지 1:1 대응 표
4. Validation: 수준 1~3 (구성요소 → 현상 → 교차 검증)
5. Patient Diversity: 4개 환자군에서 파라미터 변경 → 병리적 보행 재현
6. Discussion: Real-to-Sim 갭, 한계(상수 R, rigid tendon 근사)

**차별점**: MyoSuite는 개별 근육 파라미터가 비침습 측정 불가능. 우리는 관절 수준으로 lumping하여 등속성 검사, 진자 검사 등 표준 임상 측정으로 직접 규명 가능하다는 점이 핵심 기여.

---

#### 논문 B — 환자 파라미터화 논문 (Clinical Translation paper)
**제목 방향**: "Real-to-Sim Transfer of Pathological Neuromuscular Dynamics via Sequential Clinical Measurement"

**타겟**: IEEE TNSRE, Journal of NeuroEngineering and Rehabilitation

**핵심 주장**:
4단계 순차적 분리 규명(수동역학→최대힘→활성화→반사)이 전체 근골격 모델을 동시에 피팅하는 것보다 실현 가능하고, 각 단계가 독립적으로 해석 가능하다.

**이 논문의 전제 조건**: Role B의 Real-to-Sim 파이프라인 구현 + 실제 환자 데이터 (임상 협력 필요).

---

#### 논문 C — 웨어러블 로봇 제어 논문 (Application paper)
**제목 방향**: "Patient-Specific Exoskeleton Control via Musculoskeletal Simulation: From Clinical Measurement to Personalized Assistance"

**타겟**: ICRA, IROS, Science Robotics

**핵심 주장**:
병원에서 측정한 환자 데이터 → CALM 파라미터화 → RL 제어기 학습 → 환자별 맞춤 보조 전략의 end-to-end 파이프라인을 시뮬레이션에서 실증한다.

**이 논문의 전제 조건**: 논문 A + 논문 B + Role C~D 통합 완료.

---

### 6.2 논문 A를 쓰기 위해 지금 당장 필요한 것

| 항목 | 현재 상태 | 필요한 것 |
|------|---------|---------|
| Hill 수식 + IsaacGym 통합 | ✅ 완료 | — |
| 환자별 분화 (3종) | ✅ 완료 | 4번째 환자군 추가 |
| l_opt 파라미터 수정 | ❌ 미완 | YAML 수정 (1일) |
| 아킬레스건 SE | ❌ 미완 | 구현 필요 |
| 검증 수준 3 | ❌ 미완 | Role B와 함께 |
| 보행 기준선 | ❌ 미완 | Role D와 함께 |
| 비교 대상 (MyoSuite, OpenSim) | ❌ 없음 | 실험 설계 필요 |

**최단 경로 논문 A 제출 타임라인 (팀 구성 후 기준)**:
- Month 1~2: Phase 1 (독립 개발) + l_opt 수정
- Month 3~4: Phase 2 통합 + 검증 수준 3
- Month 5~6: 보행 기준선 + 환자 비교 실험
- Month 7~8: 논문 작성 + 제출

### 6.3 논문 포지셔닝 지도

```
          생리학적 충실도
               ↑
    MyoSuite   │  (개별 근육, 비침습 불가)
               │
    OpenSim    │  (전통 MSK, GPU 병렬 불가)
               │
  ─────────────┼──────────────────────────→ 파라미터 관측가능성
               │          CALM ★
               │          (관절 수준, 임상 측정 직접 대응)
  IsaacGym PD │
  (토크 직접)  │
```

CALM이 기존 도구들이 차지하지 못한 "충분한 생리학적 충실도 + 임상 파라미터 관측가능성 + GPU 병렬 RL" 교차점에 위치한다는 것이 논문의 핵심 포지셔닝이다.

## 7. Role별 상세 기술 명세

#### A-1. l_opt 단위 수정 (1~2일, 최우선)

**문제**: 현재 `config/healthy_baseline.yaml`의 모든 근육에 `l_opt: 1.0` (무차원)으로 되어 있다. `muscle_model.py`의 `compute_muscle_length()`는 `l_norm = muscle_length / l_opt`로 정규화하는데, `muscle_length`는 `moment_arm.py`에서 미터 단위로 계산된다. 따라서 현재 l_norm이 ~0.3 수준으로 떨어져 active force가 최대의 30%에 불과하다.

**해야 할 일**:

```yaml
quadriceps_L:
  l_opt: 0.099    # m (vastus lateralis, ~9.9cm)
hamstrings_L:
  l_opt: 0.109    # m (semimembranosus, ~10.9cm)
soleus_L:
  l_opt: 0.044    # m (~4.4cm)
gastrocnemius_L:
  l_opt: 0.051    # m (~5.1cm)
tibialis_ant_L:
  l_opt: 0.068    # m (~6.8cm)
hip_flexors_L:
  l_opt: 0.100    # m (iliopsoas, ~10cm)
gluteus_max_L:
  l_opt: 0.142    # m (~14.2cm)
```
- 좌우 동일 값 사용. 출처: Rajagopal 2016 Table 2 또는 OpenSim gait2392 참고.
- 수정 후 `validation/01_muscle_layer/test_muscle_model.py` 재실행하여 l_norm이 0.8~1.2 범위인지 확인.

**수락 기준**: 정상 보행 중간 입각기(mid-stance)에서 soleus l_norm이 0.9~1.05 범위.

#### A-3. R(q) 각도 의존성 구현 (2~4주)

**문제**: 현재 `moment_arm.py`의 R 행렬은 상수. 실제 moment arm은 관절 각도에 따라 ±30% 변한다.

**해야 할 일**: `core/moment_arm.py`를 확장하여 각도 의존 R 구현.

```python
# moment_arm.py 확장 계획

class MomentArmMatrix:
    # 현재: self.R = (num_muscles, num_dofs) 상수 텐서
    # 목표: self._compute_R(dof_pos) → (num_envs, num_muscles, num_dofs)

    def _compute_R(self, dof_pos: Tensor) -> Tensor:
        """
        방법 1 (우선): polynomial fitting
          r_i(q_j) = a0 + a1*q_j + a2*q_j^2
          계수는 OpenSim gait2392에서 추출 (스크립트로 자동화)

        방법 2 (후속): 해부학적 좌표계 기반 수치 미분
          ∂l_m/∂q_j 를 forward difference로 계산

        현실적으로 Phase 1에서는 방법 1의 선형 근사 (a0 + a1*q_j)만 구현하고,
        Phase 2에서 2차항 추가.
        """
        pass

    def compute_muscle_length(self, dof_pos: Tensor) -> Tensor:
        R = self._compute_R(dof_pos)  # (num_envs, num_muscles, num_dofs)
        # einsum 또는 bmm으로 (num_envs, num_muscles) 출력
        return l_slack - torch.einsum('emj,ej->em', R, dof_pos)
```

**필요한 사전 작업**:
- OpenSim gait2392 모델에서 각 근육-관절 쌍의 moment arm vs angle 데이터 추출
- `tools/extract_opensim_moment_arms.py` 스크립트 작성 (Python OpenSim binding)
- 추출된 데이터로 polynomial 계수 피팅 → `config/moment_arm_poly.yaml` 저장

**수락 기준**: 무릎 굴곡 0~90°에서 hamstrings moment arm이 문헌값(~4~6cm) 범위.

### Role B — 환자 파라미터화 상세 명세

#### B-1. Real-to-Sim 규명 파이프라인 구현 (핵심, 3~4주)

**목표**: 임상 측정 데이터 (CSV/MAT) → CALM 파라미터 YAML 자동 생성 파이프라인.

**구현할 스크립트 목록**:

```
tools/param_identification/
  ├── step1_passive_dynamics.py   # 수동역학 규명
  ├── step2_max_force.py          # 최대 근력 규명
  ├── step3_activation.py         # 활성화 동역학 규명
  ├── step4_reflex.py             # 반사 파라미터 규명
  ├── run_pipeline.py             # 4단계 순차 실행
  └── utils/
      ├── data_loader.py          # CSV/MAT 파일 로더
      └── optimizer.py            # scipy.optimize wrapper
```

**Step 1 — 수동역학 규명** (`step1_passive_dynamics.py`):
```
측정: Wartenberg pendulum test (진자 검사)
  - 대상자 누운 상태, 다리를 자유낙하시켜 각도·각속도 기록
  - 측정 장비: IMU 또는 VICON
입력: [시간, 각도(rad), 각속도(rad/s)] 시계열

규명 대상: 인대/관절낭 파라미터
  - k_lig: 관절 스티프니스 (N·m/rad)
  - damping: 관절 댐핑 (N·m·s/rad)
  - soft_limit_margin: 관절 가동범위 끝 soft limit

알고리즘:
  1. LigamentModel.compute_torque()를 ODE로 감쌈
  2. scipy.optimize.minimize (Nelder-Mead 또는 L-BFGS-B)
  3. 비용함수: MSE(시뮬 각도, 측정 각도)
  4. 출력: 최적 [k_lig, damping] → YAML 저장
```

**Step 2 — 최대 근력 규명** (`step2_max_force.py`):
```
측정: 등속성 근력 검사 (isokinetic dynamometry)
  - 예: Biodex System, Cybex — 표준 임상 장비
  - 측정값: 각 관절에서 최대 토크 (N·m) vs 각속도 (deg/s)
  - 보통 60, 120, 180, 240 deg/s에서 측정

규명 대상: f_max (각 근육군)
알고리즘:
  1. 측정 토크 = R × F_max × FL(l) × FV(v)
     단, FL ≈ 1.0 (등속성 검사는 최적 길이 부근에서 진행), FV는 Hill 수식
  2. 역산: F_max = torque_measured / (R × FV(v))
  3. 여러 각속도 데이터를 평균하여 f_max 추정
  4. 좌/우 비대칭 → 각각 독립 규명

주의: 한 관절에 여러 근육이 걸리면 분리 불가능 → 근육군 합산으로 처리 (현재 구조와 일치)
```

**Step 3 — 활성화 동역학 규명** (`step3_activation.py`):
```
측정: EMG + 토크 동기화 측정
  - 피부 표면 전극 (SENIAM 표준 위치)
  - EMG onset → 토크 rise 사이 지연 = tau_act 근사값
  - 자발적 이완 → 토크 decay = tau_deact 근사값

규명 대상: tau_act, tau_deact (각 근육군)
알고리즘:
  1. EMG 신호 → linear envelope (RMS, 50ms window)
  2. 정규화 EMG → u(t) (neural command proxy)
  3. ActivationDynamics ODE로 a(t) 시뮬
  4. a(t) × f_max ≈ torque(t) / R — 비교
  5. 비용함수 최소화: MSE(a_sim, a_from_torque)

현실적 제약: tau_act는 15~80ms 범위 제약 (생리학적 한계)
```

**Step 4 — 반사 파라미터 규명** (`step4_reflex.py`):
```
측정: tendon tap reflex 검사 또는 수동 신전 속도 조절 실험
  - H-reflex (전기 자극) → 반사 gain 정량화
  - 경직 환자: 수동 신전 → 반사 토크 측정

규명 대상: stretch_gain, stretch_threshold, gto_gain (RefexController 파라미터)
알고리즘:
  1. 알려진 신전 속도 프로파일로 reflex controller 시뮬
  2. 출력 activation → 토크 예측
  3. 측정 토크와 비교, 파라미터 최적화
  4. 경직(spastic): stretch_gain 높음 (1.5~3.0)
     정상: stretch_gain 1.0
     이완(flaccid): stretch_gain 낮음 (0.1~0.5)
```

```python
# Before: gymapi.acquire_gym() + gym.create_sim(PhysX params)
# After:  SimulationCfg(physics_engine=PhysicsEngine.PHYSX, physx=PhysxCfg())

# 3. 액추에이터 토크 적용
# Before: gym.set_dof_actuation_force_tensor(sim, forces)
```

1. 인간 모델만 Isaac Lab으로 먼저 이식 (Exo 없이)

4. 전체 인간+Exo 시뮬 동작 확인
