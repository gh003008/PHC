[260226 VIC01 설정 및 제어 방식 관련 문답]

본 문서는 VIC01 구현 과정에서 발생한 주요 질문들과 그에 대한 답변을 정리합니다.

1. PD 제어의 주체 관련 (Isaac Gym vs 환경 코드)
질문: 원래 V4에서는 이바단지 내부에서 제어가 이루어진 건가요? 환경 코드가 돌리는 줄 알았는데 어떤 차이가 있나요?

답변: 네, 기존 방식과 현재 방식에는 차이가 있습니다.
- 기존 V4 (control_mode: "isaac_pd"): 이 방식에서는 Isaac Gym의 물리 엔진(PhysX)이 내부적으로 PD 제어를 수행합니다. 파이썬 환경 코드는 단순히 매 스텝마다 관절의 목표 위치만 시뮬레이터에 전달하고, 실제 토크 계산은 시뮬레이터 내부에서 미리 정의된 강성(Stiffness) 값을 바탕으로 처리됩니다.
- 현재 VIC (control_mode: "pd"): 이 설정을 통해 이삭짐의 내장 PD 기능을 끄고, 파이썬 코드의 _compute_torques 함수에서 직접 토크를 계산합니다. 계산된 토크값은 힘(Force/Effort)의 형태로 시뮬레이터에 전달됩니다.

2. 제어 주체 변경에 따른 제어 주기 영향
질문: PD 제어 담당 위치가 바뀌면 제어 주기도 달라질 수 있지 않나요?

답변: 제어 주기는 동일하게 유지됩니다.
PHC의 물리 루프(physics step)는 내부적으로 서브스텝을 돌며 시뮬레이션을 진행합니다. 수동으로 구현한 _compute_torques 함수 역시 이 물리 루프 내부에서 매 서브스텝(60Hz)마다 호출되도록 설계되었습니다. 따라서 제어의 빈도나 물리적 응답성은 기존 내장 PD 방식과 차이가 없도록 안정성을 확보했습니다.

3. V4 대비 학습 곡선이 느린 이유
질문: V4는 100~200 에폭부터 보상이 크게 증가했는데, VIC01은 1000 에폭 정도부터 증가하기 시작합니다. 왜 이런 차이가 발생하나요?

답변: 주요 원인은 다음과 같습니다.
- Action Space의 확장: 정책이 결정해야 할 Action의 개수가 2배(관절 위치 + 가변 강성 계수)로 늘어났습니다. 비록 초기 단계에서 강성을 고정하더라도, 신경망의 출력 차원이 커졌기 때문에 초기 가중치를 안정화하고 유의미한 탐색을 시작하기까지 훨씬 더 많은 데이터와 학습 시간이 소요됩니다.
- 제어 복잡도: 내장 PD는 물리 엔진과 최적화되어 결합되어 있어 초기 수렴이 매우 빠릅니다. 반면 수동 토크 제어는 정책이 관절을 제어하는 방식이 물리 엔진에 직접적으로 영향을 미치므로, 보상을 얻기 시작하는 임계점까지 도달하는 데 더 신중한 학습과 시간이 필요합니다.

4. 코드 작성 방식
질문: 코드를 새로 만들 때 기존 것을 복사해서 수정한 건가요?

답변: 네, 검증된 기존 PHC 환경 로직을 유지하기 위해 원본 스크립트를 복사하여 활용했습니다.
원본 humanoid_im.py 를 humanoid_im_vic.py 로 복사한 후, 클래스 이름을 HumanoidImVIC로 변경하고 가변 임피던스 제어에 필요한 핵심 함수들(__init__, get_action_size, _compute_torques)만 오버라이드하여 수정하는 방식을 택했습니다.

5. 가변 강성 학습 전략 (커리큘럼)
질문: 초기 단계(Stage 1)는 몇 에폭까지 진행되나요? 내가 직접 종료해줘야 하나요?

답변: 네, 현재는 사용자가 학습 상태를 보고 직접 다음 단계로 넘어가는 시점을 결정하는 방식입니다. 보행의 안정성과 Reward의 수렴 정도를 모니터링하여, 기준 강성(V4 수준)만으로도 충분히 안정적으로 걷는다고 판단될 때(약 1000~2000 에폭) Stage 2 설정을 활성화하여 본격적인 강성 학습을 시작하면 됩니다.

6. 모델 형식 관련 (URDF 변환)
질문: 현재 MJCF 기반인데 URDF 모델로 변환해서 작업하는 것도 간단한가요?

답변: SMPL 모델과 같이 해부학적 구조가 복잡한 인체 모델의 경우, Isaac Gym은 MJCF 형식을 더 안정적으로 지원하고 최적화되어 있습니다. URDF로의 변환 자체는 가능하지만, 관절 좌표계 설정이나 충돌 메쉬 처리 과정에서 예상치 못한 오차가 발생할 확률이 높고 과정이 번거롭습니다. 따라서 현재의 MJCF 기반 시스템을 유지하며 제어 로직을 고도화하는 것이 가장 효율적인 방법입니다.
